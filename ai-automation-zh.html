Task Router Agent
基于 Markdown 的状态机编排层

我设计了一套基于 LLM 的任务编排运行时——路由、执行、自愈全部自动完成。一个 Markdown 文件成为 Agent 的控制平面。整套系统独立于任何活跃会话运行——编辑文件，工作就在后台发生。本文讲述：这个架构是怎么设计的，我做了哪些取舍，实际数据如何，以及一个我没编程但系统自发出现的关键行为。

第一部分

问题

背景：我作为 solo 运营者同时推进 4 条线（内容生产、首位客户的架构评审需求、作品集搭建、学业）。任务状态散落在不同工具里；每次切换上下文都要花 15–30 分钟"回血"才能真正开始工作。

产品约束：我要一个任务系统，能让我随时加任务，而分配给自动化的任务能自己执行完——不需要我守在电脑前。合上笔记本，几小时后回来，发现任务已经做完。

为什么这是 Agent 设计的关键：大多数"AI 助手"关上聊天窗口就停下。它们绑定的是 session，不是工作本身。构建一个独立于会话生命周期的 Agent 运行时，是一个根本不同的设计问题——也是 Agent 从"工具"变成"基础设施"必须做的转变。

核心设计问题：一个纯文本文件，加一套状态机协议，加一个 LLM 决策层，能不能作为一个持久化的 Agent 运行时？如果可以，这套模式就能泛化到任何运营工作流——客服 triage、内容流水线、业务自动化。

第二部分

系统架构

Task Routing 流程

触发 · task-board.md 被编辑（任何来源：我、另一个 agent、cron）
↓  fswatch · 2 秒 debounce · launchd 每小时兜底
Quiescence Gate · 30 秒文件空闲后才执行（防止编辑过程中的竞态条件）
↓  获取文件锁 · mkdir 原子操作
LLM 决策层 · 解析 task-board 状态机 · 对每个任务分类：ready / blocked:* / running:* / done / failed:*
↓
ready → 并行执行（最多 3 个 subagent 同时跑）
blocked / done → 直接跳过，不重复判断
running > 30min → 自动重置为 ready（死锁恢复）
↓  单写者协议 · 主 agent 统一写回状态
状态更新 + 只追加日志 · task-board.md 写回 · events.jsonl 结构化日志 · deadletter/ 终态失败归档

5 层架构

L3执行层bash + claude -p无状态一次性调用
L2决策层task-check skill (LLM)状态机路由 + 任务交接

第三部分

关键设计决策

4 个深思熟虑的取舍。每个都有我明确拒绝的替代方案。

1. 用 Markdown 做控制平面，不用 Notion / Airtable / SQL
拒绝结构化数据库（Notion、Airtable、Postgres）——schema 强制、可查询、天然支持多用户
选择一个 Markdown 文件 + 每个任务的 YAML 式元数据
理由编辑延迟 < 1 秒，零厂商依赖，git 可版本化，无工具即可读，LLM 无需 schema inference 即可解析。代价——缺 schema 约束——由状态机层（见决策 2）补足。
何时重审多人并发编辑成为硬需求，或任务量超 ~500 条 Markdown 解析变瓶颈时
2. 类型化状态机，不用自然语言 status
拒绝自由文本状态（"等客户回复"、"进行中"、"卡在 API"）
选择类型化状态 token：ready / blocked:siyao:send-brief / running:agent-a4b:2026-04-20T15:25Z / done / failed:exhausted_retries
理由类型化状态是协议，不是标签。LLM 解析它时能推理依赖链、死锁条件、时间戳合理性——不只是字符串 pattern match。对系统"智能度"的最大单点影响（见第五部分）。
代价一次性迁移所有现有任务（13 张卡）+ 维护 taxonomy 的纪律。前期成本小，后期复利
3. 单写者并发，不用分布式锁
拒绝多 agent 并发写 task-board.md + 分布式锁（flock、append 模式、CRDT）
选择严格单写者协议：只有主 agent 能写状态。Subagent 并行执行任务 + 返回结构化结果；主 agent 串行化所有状态更新
理由Markdown 文件不是 transactional 的。并发写会丢失更新、结构损坏、或产生无法解析的状态——一套连锁故障。单写者限制了并行带宽，但以零运行时成本消除了一整类 race condition bug。
取舍执行并行性保留（最多 3 subagent 并行）；状态串行化只在任务吞吐量持续 > 1/s 时成为瓶颈。这不是当前工作量的真实约束。
4. Quiescence Gate，不用简单 debounce
拒绝文件 watcher 固定 2 秒 debounce
选择Watcher debounce（2s）叠加 quiescence gate：执行前必须有连续 30 秒文件空闲。每 5 秒轮询一次；任何改动即重置计时。
理由用户一次"保存"动作，编辑器可能触发 watcher 5-10 次。没有 quiescence 的话，执行器会和用户抢时间；状态更新会和还在进行的编辑撞车。30 秒 gate 让自动化处理和用户的自然停顿节奏对齐。
代价处理延迟最低 30 秒。对异步任务流可接受；不适合实时聊天 agent。

第五部分

Emergent Behavior · 核心洞察

我给决策层的编程只有一件事：找 ready 状态的任务，执行它们。其他一律跳过。

首次生产运行时，它做了我没有编程的事：

  • 捕捉到一个我自己写的语义不一致：一个任务标为 blocked:task:TASK-015，但 TASK-015 已经是 done。skill 标记"依赖链断了"，建议把它升为 ready，并问我这是否符合我的意图。
  • 识别了一个 scope 模糊：我标为 ready 的任务，skill 对比 Measure 字段和可用输入，注意到一个必须的产出物（portfolio URL）缺失，拒绝执行——等我澄清。
  • 发现了一个时区 bug：我手动写的 running:claude-a:2026-04-20T15:25Z 实际是本地时间被错标为 UTC。skill 计算出这个时间戳比 now 领先 2 小时，标为 anomaly 问我确认。

这些行为都不在 prompt 里。它们从状态机设计中涌现出来。

同一个 LLM，同一个 task-board，agent 的行为完全不同——取决于 status 是标签还是协议。自然语言 status（"等待"、"进行中"）给模型一个 token 去匹配。类型化 status + 结构化子字段给模型一个协议去推理。我给产品团队打包的核心 takeaway：让 agent 从"执行机器"变成"推理搭档"的不是更好的 prompt，是结构化的状态。

这对 agent 系统设计有具体含义：大多数团队在 agent 行为不满意时会去迭代 prompt。更高杠杆的动作往往是改变 agent 运作所在的类型系统。

第六部分

泛化与下一步

模式迁移

这套架构不是任务自动化专用的。同样的 5 层模式（控制平面文件 + 事件驱动调度 + 无状态执行器 + 类型化状态机 + 只追加审计日志）可以直接对应到一位 Shopify 客户在简报里提出的几类工作流——我正在为这份简报做架构评审：

工作流控制平面决策层任务
客服工单分流Helpdesk ticket 队列分类 + 起草回复 + 标记待审核
预售订单管理供应商 ETA 追踪表识别变动 + 生成客户通知
供应商数据产品创建接收表格队列字段抽取 + 标准化 + 写入草稿
Collection-buying 收品咨询邮件日志抽取 + 评分 + 起草跟进

同一套状态机规范（ready / blocked:* / running:* / done）适用所有。workflow-specific 的只有决策层的 prompt 模板。基础设施共享。

规划演进

  • Observability 界面：通过 launchd 提供本地 HTML dashboard，展示任务吞吐量、错误率、延迟分布随时间的变化
  • Prompt 版本控制：决策层 prompt 独立进 git 仓库 + 语义化版本 + 基于捕获输入语料的 regression 测试
  • 多 agent 交接协议：标准化结果 schema，让任务独立时 subagent 可以互相 chain 而不经主 agent 中转
  • 状态持久化升级：如果跨工作流查询成为反复需求，在 Markdown 旁边加 SQLite（Markdown 仍作人类 UI）

下次我会怎么做

  • 第一天就 instrument metrics，不要拖到第七天。events.jsonl 里哪怕是粗糙的 counter，都比事后加的精确 metric 值钱——追溯测量永远是估算
  • 先在纸上画状态机，再写代码。我是在运行中遇到解析歧义才从自然语言状态迁移到类型化状态。正确顺序是：纸上状态机 → skill 设计 → 代码实现。我做反了。

搭这个系统让我带走一条可以用在任何 agent 产品上的经验：状态机才是产品。Prompt、subagent、调度层——都是可替代的。人类意图和 agent 行动之间的状态契约，才是那个会持久化、会规模化、决定"系统能推理还是只能执行"的东西。
